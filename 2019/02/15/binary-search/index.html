No.162 Find Peak Element

https://leetcode.com/problems/find-peak-element/

analysis

The brute force is so easy that we just traverse every element in nums and compare them with two elements locating beside them. This will lead to $O(n)​$. To lower it to $O(logn)​$, We can use banary search to solve this problem. When we find the middle element is exactly the peak, we return it. Otherwise we need to observe the tendency of the growth or shrinking. If the tendency is growing, the peak must on the right side, do we go recursion on the right side. For shrinking tendency, vice versa. If there is no tendency, we need to check both side to find the peak.

Follow up for Leetcode No.162

description

Based on the problem No.162, the follow up problem is to count the number of all the peaks. We can also imagine that the left side and the right side is infinite negative integer. If input array is sorted in strictly increasing order, the last element is always a peak element. For example, 50 is peak element in {10, 20, 30, 40, 50}. If input array is sorted in strictly decreasing order, the first element is always a peak element. 100 is the peak element in {100, 80, 60, 50, 20}. If all elements of input array are same, every element is a peak element. Also, peaks can be consecutive.

analysis

If we need to count all the peaks, we can also use binary search. The difference is that when we find a $mid$ position which is a peak, we add $1$ to result value. Then check the left side and the right side. Also, importantly, to avoid duplication. We introduce a $visited$ array to remember all the number we have already checked. The running time is $O(logn)​$.
