No.121 Best Time to Buy and Sell Stock

https://leetcode.com/problems/best-time-to-buy-and-sell-stock/

analysis

The brute force algorithm is use two loops to check every possible profit since at most one buy and sell transaction is allowed. It’s a $O(n^{2})$ algorithm, but will lead to TLE.

Then we think about another algorithm, we only care about two stock transaction time. One is to buy the stock $b_{i}$, and the other one is to sell the stock $s_{i}$. Only if $s_{i}-b_{i} \geq 0$, we can gain profit from the transaction. So we just traversal every stock and update the lowest stock price and biggest profit simultaneously. We can get the biggest profit at last. It lead to $O(n)​$.

Greedy Algorithm $O(n)​$

No.122 Best Time to Buy and Sell Stock II

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-ii/

analysis

Following the thought in No.121, firstly we can also initialize two variables to memorize the lowPrice and highPrice. One important thing in this algorithm is that it is possible for the loop to stop without updating the maxProfit because the last element of the input is larger than the previous one, so we would not go to another $if$ branch. For example, if the imput is $[7,6,5]$, then the ouput may be $0$. So we have to judge this condition every moment we find a more bigger stock. It leads to $O(n)$. This algorithm is extremelt slow. We have to improve it by decreasing some $if$ branches.

Then it came out that we can only focus on the difference between $prices[index+1]$ and $prices[index]$, because the continuous increment result is definitely what we will get using highPrice and lowPrice. It is also an $O(n)$ algorithm, but much faster.

code

Keep updating highPrice and lowPrice $O(n)$ (slow)

Focus on $prices[index+1]>prices[index]$ $O(n)​$ (faster)

No.123 Best Time to Buy and Sell Stock III

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-iii/

analysis

The first algorithm is using DP with state memorization. There are $4$ states in the transaction process, which are $buy1$, $buy2$, $sell1$ and $sell2$. And in different states, we can get different accumulative profit. So we use $4$ arrays to memorize the accumulative profit in different states during traversal and update them according to their transaction rule and $occurring$ $order$. For $buy1$, if current stock is the cheaper than previous stocks, then we update $buy_1[i]$ to $-prices[i-1]$, otherwise using $buy_1[i-1]$. For $sell1$, we need to keep the maximum of the $sell_1[i-1]$ (not sell this time) and $buy_1[i-1] + prices[i-1]$ (sell this time). For $buy2$, we need to keep the maximum of $buy_2[i-1]$ (not nuy this time) and $sell_1[i-1] - prices[i-1]$ (buy this time). For $sell2$, we need to keep the maximum of $sell_2[i-1]$ (not sell this time) and $buy_2[i-1] + prices[i-1]$ (sell this time). To memorize these states, we need to use $O(n)$ to store arrays. It leads to $O(n)$ running time.

The second algorithm is a more universe one which can be used to $k$ transaction numbers. We define a $(k+1)len(prices)$ matrix to memorize the DP values. we will traverse the prices for $k$ times to emulate the $k$ transaction process. In each transaction process, we update the DP matrix and maxProfit as well. We need to keep the maximum of the current profit between to sell or not to sell and to buy or not to buy. It leads to $O(kn)$.

code

Dynamic Programming with state, $O(n)​$ in time and $O(n)​$ in space

Dynamic Programming with matrix, $O(kn)$ in time and $O((k+1)n)$ in space, universality for $k$

We solve this in $O(kn)​$.

No.309 Best Time to Buy and Sell Stock with Cooldown

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-with-cooldown/

analysis

From the problem statement, there should be 3 states in the transaction process: $buy$, $sell$, $cool$. The $cool$ state can be transfered from $sell$ (cool this time) and $cool$ (cool again, more than one times). The $buy$ state can be transferred from $cool$ (buy this time) and $buy$ (not buy this time). The sell state can be transferred from $buy$ (sell this time) and $sell$ (not sell this time). It leads to $O(n)$.

code

DP with state memorization, $O(n)$ in time

No.621 Task Schedule

https://leetcode.com/problems/task-scheduler/

analysis

We can solve thos problem based on observation and mathematics. For example, given $[‘A’, ‘A’, ‘B’, ‘B’, ‘C’, ‘C’, ‘C’ ]​$ and $2​$, it is obvious that the interval numbers should at least have a length of $(maxNum-1)(n+1)​$ before the last $C​$ so that we can schedule all the $C​$ tasks. And for the last $C​$, we just add one more interval to the result, then return. Then another example is $[‘A’, ‘A’, ‘B’, ‘B’, ‘B’, ‘C’, ‘C’, ‘C’ ]​$ and $2​$. The numbers of $B​$ and $C​$ are both the largest number, so we need to add two more intervals to the result. So we conclude that we can calculate the number of all the tasks occur the most frequently and add the number to the result.

Another important thing is a tricky example, such as $[‘A’, ‘A’, ‘B’, ‘B’, ‘C’, ‘C’]$ and 0. We may get $1$ following the previous algorithm, but actually we should return $6$. This occurs because the number of different tasks is more than $n+1$, so we don’t need to considering adding $idle$ interval.

Overall, return the maximum of $len(tasks)$ and the result from the previous algorithm should be reasonable. It leads to $O(n)$.

No.207 Course Schedule

https://leetcode.com/problems/course-schedule/

analysis

This is a typical graph problem. The prerequisites array forms a directed graph. Here we use breath-first-search to solve the problem. We at first establish a graph mapping the courses and their prerequisites. Then we also need to know how many times each course are taben to be a prerequisite for others. If a course which are not taken by any other courses to be a prerequisite, that means it’s free for us to take that course. For those classes which are not prerequisites, we will traverse them and update the courseNum simultaneously. During the traversal, we need to find their prerequisites and reduce the number by $1$ for each prerequisite. If the number is reduced to $0$, we can add this course to the deque, which means no course should take this course as prerequisite any more. At last, we need to campare $courseNum$ and $numCourses$ and return the result. It leads to $O(n)$.

code

BFS using deque, $O(len(prerequistes))​$ (the edge of graph)

No.630 Course Schedule III

https://leetcode.com/problems/course-schedule-iii/

analysis

This is a typical greedy alforithm problem, which is a little similar to interval scheduling. So similarly we need to sort the courses by their ending time. Then we traversal the courses, and everytime if we find that the accumulative starting time is higher than the end, we need to reduce the courses with the longest lasting time so that we can gain more time to add more new courses.

Here we use a heap to maintain a priority queue. In python, we can use $heapq$ to maintain a min heap. And we can transfer our min heap to ‘max heap’ using negative numbers. Everytime we traverse the courses, we push the nehative number of lasting time to the heap. As soon as we need to delete a course, we just pop the minmum element from the heap. Since we use negative numbers before, we can get the biggest positive lasting time.

It leads to $O(n)$.

This problem is also possible to be solved using DP.

code

Greedy Algorithm with priority queue (heap), $O(n^{2})​$

No.56 Merge Intervals

https://leetcode.com/problems/merge-intervals/

analysis

Intercal problem can always be solved through greedy algorithm. Here we also use greedy algorithm to solve it. But we cannot use the ending time to sort the intervals again like classic interval schedule problem. Because in classic interval schedule problem, we need to accept as more intervals as possible. But here we need to get as less intervals as possible. So here we need to focus on the starting time as well and to the priority. So we first sort the intervals by their ascenging starting time. Then traverse the intervals, if we find a interval whose starting time is lower or equal to the current ending time, we need to update current ending time. If not, that means we can not merge it to the previous interval, so we just add it into result array and update starting time and ending time.

It leads to $O(n)$.

code

Greedy Algorithm $O(n)$

If not we will add $interval$ to the result array, as well as the $newInterval$ under the condition that $newInterval.end<interval.start$ and the newInterval has not been added before.
