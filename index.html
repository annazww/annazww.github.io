No.121 Best Time to Buy and Sell Stock

https://leetcode.com/problems/best-time-to-buy-and-sell-stock/

analysis

The brute force algorithm is use two loops to check every possible profit since at most one buy and sell transaction is allowed. It’s a $O(n^{2})$ algorithm, but will lead to TLE.

Then we think about another algorithm, we only care about two stock transaction time. One is to buy the stock $b_{i}$, and the other one is to sell the stock $s_{i}$. Only if $s_{i}-b_{i} \geq 0$, we can gain profit from the transaction. So we just traversal every stock and update the lowest stock price and biggest profit simultaneously. We can get the biggest profit at last. It lead to $O(n)​$.

Greedy Algorithm $O(n)​$

No.122 Best Time to Buy and Sell Stock II

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-ii/

analysis

Following the thought in No.121, firstly we can also initialize two variables to memorize the lowPrice and highPrice. One important thing in this algorithm is that it is possible for the loop to stop without updating the maxProfit because the last element of the input is larger than the previous one, so we would not go to another $if$ branch. For example, if the imput is $[7,6,5]$, then the ouput may be $0$. So we have to judge this condition every moment we find a more bigger stock. It leads to $O(n)$. This algorithm is extremelt slow. We have to improve it by decreasing some $if$ branches.

Then it came out that we can only focus on the difference between $prices[index+1]$ and $prices[index]$, because the continuous increment result is definitely what we will get using highPrice and lowPrice. It is also an $O(n)$ algorithm, but much faster.

code

Keep updating highPrice and lowPrice $O(n)$ (slow)

Focus on $prices[index+1]>prices[index]$ $O(n)​$ (faster)

No.123 Best Time to Buy and Sell Stock III

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-iii/

analysis

The first algorithm is using DP with state memorization. There are $4$ states in the transaction process, which are $buy1$, $buy2$, $sell1$ and $sell2$. And in different states, we can get different accumulative profit. So we use $4$ arrays to memorize the accumulative profit in different states during traversal and update them according to their transaction rule and $occurring$ $order$. For $buy1$, if current stock is the cheaper than previous stocks, then we update $buy_1[i]$ to $-prices[i-1]$, otherwise using $buy_1[i-1]$. For $sell1$, we need to keep the maximum of the $sell_1[i-1]$ (not sell this time) and $buy_1[i-1] + prices[i-1]$ (sell this time). For $buy2$, we need to keep the maximum of $buy_2[i-1]$ (not nuy this time) and $sell_1[i-1] - prices[i-1]$ (buy this time). For $sell2$, we need to keep the maximum of $sell_2[i-1]$ (not sell this time) and $buy_2[i-1] + prices[i-1]$ (sell this time). To memorize these states, we need to use $O(n)$ to store arrays. It leads to $O(n)$ running time.

The second algorithm is a more universe one which can be used to $k$ transaction numbers. We define a $(k+1)len(prices)$ matrix to memorize the DP values. we will traverse the prices for $k$ times to emulate the $k$ transaction process. In each transaction process, we update the DP matrix and maxProfit as well. We need to keep the maximum of the current profit between to sell or not to sell and to buy or not to buy. It leads to $O(kn)$.

code

Dynamic Programming with state, $O(n)​$ in time and $O(n)​$ in space

Dynamic Programming with matrix, $O(kn)$ in time and $O((k+1)n)$ in space, universality for $k$

No.188 Best Time to Buy and Sell Stock IV

https://leetcode.com/problems/best-time-to-buy-and-sell-stock-iv/

analysis

We may apply the thought in No.123 to solve the problem as well, but it will cause memory error. So we need to make some improvement on the storing space.

Firstly, we should notice that during each traversal, what definitely we need is just the DP value in last traversl, instead of the DP value in all previous traversals. We can use only two rows to memorize the DP value and update two rows in every reaversal. But actually this causes memory error again.

Then, we notive that the memory error occurs at the code line “for times in range(1,k+1)”, which implie the memory error may be related to the extremely large of $k$. And thinking about the relation of $k$ and $len(prices)$, if $k > len(prices)$, this problem can be simplified to problem No.122. If not, then we use DP to solve it. There is no memory error again. We solve this in $O(kn)​$.

code

Dynamic Programming with matrix, $O(kn)$ in time and $O(1)$ in space, universality for $k$, memory error

reduce the dp matrix to $2len(prices)$, still memory error

judge the relation of $k​$ and $len(prices)​$ at first, accepted, $O(kn)​$ in time, $O((k+1)n)​$ in space
